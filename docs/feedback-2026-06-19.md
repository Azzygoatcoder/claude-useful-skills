# Audit Skill Feedback — 2026-06-19

Source: 3 rounds of security audit on sjtu-agent (v0.4.0, v0.6.1, ChromaDB memory) using code-security-audit v1.1.1.

---

## 1. Phase 1 并行不靠谱

3 个 Agent 的 prompt 都有清晰定义，但实际执行中经常只回来 1-2 个结果。v0.6.1 审计发了三个 Agent，实际只收到一个有效报告。

**不是 skill 设计问题，是工具层面的**——但 skill 应该提供单 Agent 兜底方案：并行失败时自动降级为单 Agent 全量扫描。

## 2. 误报率 ~30-40%

Agent C 报了 7 个 finding，验证后只有 4 个成立。典型误报：
- `open_id` 覆写被标记为 unauthenticated overwrite，实际代码在白名单检查之后运行
- `import agent` 在 try/except 内被标记为 fragile deserialization，实际有 graceful degradation

Phase 2 去重步骤有作用，但"判断是否误报"仍然全靠人工读代码——这是整个流程的瓶颈。

**建议**：在 Agent prompt 中加入误报示例，要求 Agent 标注 confidence level。

## 3. Edit 工具和审计报告的摩擦

更新 `SECURITY_AUDIT.md` 时多次 `old_string` 匹配失败（Unicode/空白字符差异）。状态注解 `<!-- AUDIT:STATUS=... -->` 设计上支持 `mark-fixed` 轻量操作，但实际用起来还是手动逐行改。

**建议**：skill 在 Phase 4 或 mark-fixed 时生成一个 shell 脚本（`sed` 或 Python），批量更新注解行，避免 Edit 工具的字符串匹配问题。

## 4. "增量"没真正跑过

v1.1 设计了 `git diff <audit-commit>..HEAD` 增量 re-audit，但三次审计都是新代码范围，不是 re-audit。这个功能需要一次真正的 re-audit 来验证可行性。

## 5. 缺少"新代码自动发现"

每次审计都要手动告诉 Agent "看看 `_dining.py`、`calendar.py`、`canteen_crowd.py`"。skill 应该：
1. 读取上次审计报告的 `Audit commit` 字段
2. `git diff --name-only <last-commit>..HEAD`
3. 将变更文件列表注入到 Phase 1 的每个 Agent prompt 中

这样新代码自动获得更高审查权重。

## 6. 健康检查比较弱

没有"上次审计以来的依赖变化"检查。实际遇到了 `browser-use` 升级导致 `pypdf` 版本冲突（`pypdf>=4.0.0,<6.0.0` vs `browser-use` requires `pypdf==6.10.2`）。

Phase 1 Agent C 应该能发现这类问题，但需要明确的 prompt 指引：
- 跑 `pip check` 或对比 `requirements.txt` diff
- 检查新增依赖是否有已知 CVE

## 优先级

| # | 改进 | 影响 |
|---|------|------|
| 1 | Phase 1 自动 diff 新代码范围 | 减少人工指定文件的工作 |
| 2 | 降低误报率的 prompt 调优（加 confidence level） | 节省 Phase 2 时间 |
| 3 | 单 Agent 兜底模式（并行失败降级） | 不丢覆盖 |
| 4 | 生成 mark-fixed 批量更新脚本 | 减少 Edit 工具摩擦 |
| 5 | 依赖变更自动检查 | 防止 CI 级问题 |
| 6 | 增量 re-audit 实战验证 | 验证 v1.1 核心功能 |
